# Three.js + MediaPipe 3D 粒子交互项目架构分析

## 一、渲染管线与调用链分析

### 1.1 初始化流程

```
main.js:5-8
  ↓
App.js:10-30 (constructor)
  ├─ Renderer.js:5-26 (创建场景、相机、渲染器)
  ├─ UIManager (UI管理)
  ├─ CameraController.js:5-21 (相机控制器)
  ├─ Planet.js:27-95 (创建行星粒子系统)
  ├─ Rings.js:25-93 (创建环带粒子系统)
  ├─ GestureManager.js:4-63 (初始化MediaPipe手势识别)
  └─ 添加环境光
  ↓
App.js:32-34 (start())
  ↓
App.js:36-48 (animate() 启动渲染循环)
```

### 1.2 每帧更新调用链

**主渲染循环 (App.js:36-48)**

```javascript
animate() {
  requestAnimationFrame(this.animate.bind(this));
  const time = this.clock.getElapsedTime();
  
  // 更新逻辑
  this.cameraController.update();    // 更新相机位置
  this.planet.update(time);           // 更新行星旋转
  this.rings.update(time);            // 更新环带旋转
  
  // 渲染
  this.renderer.render();             // 渲染场景
}
```

**完整调用链：**

```
requestAnimationFrame
  ↓
App.animate()
  ├─ CameraController.update()  [core/CameraController.js:24-60]
  │   ├─ 处理缩放逻辑 (zoom in/out)
  │   ├─ 应用平滑阻尼
  │   ├─ updateCamera() → 更新相机位置
  │   └─ 更新UI显示
  │
  ├─ Planet.update(time)  [world/Planet.js:97-101]
  │   └─ mesh.rotation.y = time * 0.05 (绕Y轴旋转)
  │
  ├─ Rings.update(time)  [world/Rings.js:95-99]
  │   └─ mesh.rotation.y = -time * 0.02 (反向绕Y轴旋转)
  │
  └─ Renderer.render()  [core/Renderer.js:34-36]
      └─ renderer.render(scene, camera)
```

## 二、GestureManager 与 Three.js 交互机制

### 2.1 数据流向图

```
MediaPipe Hands
    ↓ (关键点数据)
GestureManager.onResults()  [input/GestureManager.js:65]
    ├─ 旋转检测
    │   ├─ calculateHandRotation() → 角度计算
    │   ├─ 历史平滑处理
    │   └─ CameraController.rotate(deltaTheta)
    │
    ├─ 缩放检测
    │   ├─ detectFist() / detectOpenHand() → 手势分类
    │   ├─ 计算手的大小变化
    │   └─ CameraController.startZoomIn() / startZoomOut()
    │
    └─ UI更新
        └─ UIManager.highlightCard() / updateIntensity()
```

### 2.2 关键交互点

**1. 旋转交互 (GestureManager.js:86-112)**

```javascript
// 计算手的旋转角度
const handAngle = this.calculateHandRotation(landmarks);
// ... 平滑处理后
this.cameraController.rotate(avgRotation);  // 传递给相机控制器
```

**2. 缩放交互 (GestureManager.js:114-157)**

```javascript
// 握拳 + 拉远 → Zoom Out
if (isFist && deltaSize < -moveThreshold) {
  this.cameraController.startZoomOut();
}
// 张开 + 靠近 → Zoom In
else if (isOpen && deltaSize > moveThreshold) {
  this.cameraController.startZoomIn();
}
```

### 2.3 重要特性

- **解耦设计**：GestureManager 不直接操作 Three.js 场景对象，只通过 CameraController 的公共接口交互
- **防抖机制**：连续5帧无手才停止动作，避免抖动 (GestureManager.js:69)
- **平滑处理**：旋转角度使用历史记录平均，减少抖动 (GestureManager.js:97-103)
- **状态管理**：使用 `state` 变量跟踪当前手势状态 (IDLE/ZOOM_IN/ZOOM_OUT)

## 三、Planet.js 与 Rings.js 职责划分与耦合分析

### 3.1 职责划分

| 模块 | 职责 | 核心功能 |
|------|------|----------|
| **Planet.js** | 行星球体粒子系统 | 70%表面粒子 + 30%内部粒子，橙金色渐变 |
| **Rings.js** | 环带粒子系统 | 内环+外环双层结构，白色发光粒子 |

### 3.2 Planet.js 详细职责 [world/Planet.js:27-101]

**核心特性：**
- 粒子总数：20000 个 (CONFIG.PLANET.PARTICLE_COUNT)
- 分布策略：70%在球面附近，30%在内部 (使球面更密集)
- 颜色：橙色到金色渐变 (0xff8c00 为基础)
- 纹理：自定义发光渐变纹理
- 动画：绕Y轴正方向旋转 (速度: 0.05)

**关键代码：**
```javascript
// 粒子分布策略 (Planet.js:47-76)
if (Math.random() < surfaceRatio) {
  r = radius * (0.9 + Math.random() * 0.1);  // 表面粒子
} else {
  r = radius * Math.cbrt(Math.random()) * 0.9;  // 内部粒子
}
```

### 3.3 Rings.js 详细职责 [world/Rings.js:25-99]

**核心特性：**
- 粒子总数：8000 个 (CONFIG.RINGS.PARTICLE_COUNT)
- 结构：双层环带 (50%内环 + 50%外环)
- 颜色：白色发光 (0xffffff)
- 倾斜：X轴倾斜 0.12π，Z轴倾斜 0.05π
- 动画：绕Y轴反方向旋转 (速度: 0.02)

**关键代码：**
```javascript
// 双层环带结构 (Rings.js:43-71)
if (i < count * 0.5) {
  baseRadius = innerRadius;  // 内环
  spread = 25;
} else {
  baseRadius = outerRadius;  // 外环
  spread = 35;
}
```

### 3.4 耦合性分析

**结论：两个模块完全解耦，无直接依赖**

**证据：**
1. **无相互引用**：Planet.js 和 Rings.js 互不 import
2. **独立初始化**：都只接收 scene 参数，各自创建 mesh
3. **独立更新**：update() 方法只更新自己的 rotation，互不影响
4. **无共享状态**：没有共同的变量或数据结构
5. **唯一共同点**：都使用 Three.js 的 Points 粒子系统，都被 App 统一管理

**App 中的管理方式 (App.js:18-19, 43-44)：**
```javascript
// 独立创建
this.planet = new Planet(this.scene);
this.rings = new Rings(this.scene);

// 独立更新
this.planet.update(time);
this.rings.update(time);
```

### 3.5 代码重复分析

两个模块存在**代码重复**：
- `createParticleTexture()` 函数几乎完全相同，只有渐变颜色参数不同
- 粒子系统的初始化模式相同 (BufferGeometry + PointsMaterial + Points)

**优化建议**：可以提取一个 `ParticleSystem` 基类或工具函数来减少重复代码。

## 四、整体架构总结

### 4.1 架构分层

```
┌─────────────────────────────────────────┐
│              App.js                     │  编排层：统一初始化和更新
├─────────────┬─────────────┬─────────────┤
│  Renderer   │ CameraCtrl  │ GestureMgr  │  核心控制层
├─────────────┴─────────────┴─────────────┤
│           Planet / Rings                │  场景对象层
├─────────────────────────────────────────┤
│         Three.js / MediaPipe            │  第三方库层
└─────────────────────────────────────────┘
```

### 4.2 设计优点

1. **模块化清晰**：每个类职责单一，符合单一职责原则
2. **依赖注入**：通过构造函数注入依赖 (如 scene, cameraController)
3. **接口隔离**：模块间通过公共方法交互，不暴露内部实现
4. **异步分离**：手势识别与主渲染循环分离，避免阻塞

### 4.3 数据流总览

```
用户手势 → MediaPipe → GestureManager → CameraController
                                                        ↓
                                                  Three.js Scene
                                                        ↓
                                            Planet / Rings 更新
                                                        ↓
                                                  Renderer 渲染
```
